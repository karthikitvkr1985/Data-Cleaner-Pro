## 3. Core Data Models

Define these as Pydantic models in `backend/models/schemas.py` and mirrored TypeScript types in `frontend/src/types/models.ts`. Keep them in sync exactly.

```python
class ColumnProfile(BaseModel):
    name: str
    inferred_type: Literal["string", "integer", "float", "datetime", "boolean", "categorical"]
    null_count: int
    unique_count: int
    sample_values: list[str]
    stats: dict  # min/max/mean for numeric, top values for categorical, etc.

class Suggestion(BaseModel):
    id: str
    row_index: int | None      # null for column-level suggestions
    column_name: str
    original_value: str | None
    proposed_value: str | None
    reason: str
    category: Literal["type_fix", "format", "duplicate", "missing_value", "validation", "structure"]
    status: Literal["pending", "accepted", "rejected", "edited"]

class TableDetectionResult(BaseModel):
    sheet_name: str
    tables: list[dict]   # each: {start_row, end_row, start_col, end_col, header_row, confidence}

class Session(BaseModel):
    session_id: str
    original_filename: str
    sheets: list[str]
    columns: list[ColumnProfile]
    suggestions: list[Suggestion]
    recipe_steps: list[dict]
    created_at: datetime
    # NOTE: no file bytes stored on this model persisted anywhere — file lives only
    # in an in-memory cache keyed by session_id, expired after N minutes of inactivity

class RecipeStep(BaseModel):
    step_id: str
    module: Literal["structure", "type", "format", "dedup", "missing", "validation"]
    description: str          # human-readable, e.g. "Standardize 'state' column to Title Case"
    params: dict               # machine-readable parameters to replay this step
```

## 4. API Contract

All endpoints under `/api`. JSON in/out except upload/export which use file bodies.

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/upload` | Accept file upload, return `session_id` + list of detected sheets |
| `POST` | `/api/sessions/{session_id}/detect-structure` | Run structure/table/header detection on a chosen sheet, return `TableDetectionResult` |
| `POST` | `/api/sessions/{session_id}/analyze` | Run type inference, missing-value scan, dedup scan, and generate `Suggestion[]` |
| `GET` | `/api/sessions/{session_id}/suggestions` | Return current suggestion queue (with status filters via query params) |
| `POST` | `/api/sessions/{session_id}/suggestions/{suggestion_id}` | Accept/reject/edit a single suggestion. Body: `{ "status": "accepted" \| "rejected" \| "edited", "edited_value": optional }` |
| `POST` | `/api/sessions/{session_id}/suggestions/bulk` | Bulk accept/reject by category or column. Body: `{ "category": ..., "column_name": optional, "status": ... }` |
| `POST` | `/api/sessions/{session_id}/nl-command` | Submit a natural language instruction, return a preview of proposed changes (not yet applied) |
| `POST` | `/api/sessions/{session_id}/nl-command/confirm` | Apply a previously previewed NL command |
| `GET` | `/api/sessions/{session_id}/preview` | Return current grid state (paginated/windowed for large files) |
| `GET` | `/api/sessions/{session_id}/recipe` | Return the full ordered list of `RecipeStep` taken so far |
| `POST` | `/api/sessions/{session_id}/recipe/apply-to-new-file` | Apply an existing recipe to a newly uploaded file with matching/similar schema |
| `GET` | `/api/sessions/{session_id}/export` | Return the cleaned file (`.xlsx` or `.csv`, via query param `format`) |
| `GET` | `/api/sessions/{session_id}/issues-report` | Return the Issues Report as a downloadable sheet/file |
| `DELETE` | `/api/sessions/{session_id}` | Explicitly purge session from memory (also auto-expire after inactivity) |

## 5. Feature Modules — Implementation Detail

### 5.1 Structure Detection (`engine/structure_detection.py`)
- Use pandas + openpyxl to read raw sheet grid (not `read_excel` directly first — read as raw cells to detect layout before assuming a header row)
- Heuristics: a row is a likely header if it has high text-density and the row below has a type shift (e.g., text header → numeric data)
- Detect table boundaries via runs of blank rows/columns as separators
- Detect merged cells via openpyxl's `merged_cells` and propagate the top-left value to all cells in the merged range
- Return confidence score per detected table so the UI can flag low-confidence detections for manual review

### 5.2 Type Inference (`engine/type_inference.py`)
- For each column: attempt numeric coercion, then datetime coercion (try a ranked list of common formats), then boolean coercion, fall back to string/categorical
- Categorical vs. free-text string distinction: if unique_count / total_count < 0.5 and unique_count < 50, treat as categorical
- Return per-column `ColumnProfile` with stats

### 5.3 Formatting (`engine/formatting.py`)
- Whitespace: strip + collapse internal double-spaces, strip non-printable/control characters
- Case standardization: expose as a column-level user-selectable transform (upper/lower/title/sentence), never auto-applied without suggestion approval
- Currency/number cleanup: strip symbols and thousands separators via regex before numeric coercion, log what was stripped
- Date normalization: parse to `datetime`, re-emit in one consistent ISO format unless user specifies another
- Email/phone/URL: validate against standard regex patterns, flag non-conforming values as suggestions rather than silently dropping them

### 5.4 Deduplication (`engine/deduplication.py`)
- Exact duplicates: pandas `duplicated()` on user-selected key columns (default: all columns)
- Near-duplicates: normalize (lowercase, strip, remove punctuation) into a helper column, then flag matches; use `rapidfuzz` for fuzzy string similarity (threshold configurable, default 90)
- Categorical canonicalization: cluster similar values (e.g., via `rapidfuzz.process`) and propose one canonical label per cluster, shown to the user before merging

### 5.5 Missing Values (`engine/missing_values.py`)
- Report missing rate per column
- Strategies exposed per column: mean/median/mode fill, forward-fill, custom constant, sentinel tag (e.g., `"N/A"`), or drop row
- Never apply a fill without it appearing in the suggestion/review queue first

### 5.6 Validation (`engine/validation.py`)
- Auto-generate `pandera` schema checks per column based on inferred type + observed range/allowed values
- Surface each generated rule to the user as editable/removable before enforcement
- Run checks, emit violations as `Suggestion` objects with category `validation`

### 5.7 Recipe Engine (`engine/recipe.py`)
- Every accepted suggestion or applied NL command appends a `RecipeStep` to the session
- Recipe must be replayable: given a new file with a similar (fuzzy-matched) schema, re-run the same ordered steps against the new columns
- Export recipe as JSON and as a human-readable markdown summary

### 5.8 Natural Language Commands (`ai/suggestion_engine.py` + `routers/apply.py`)
- Parse instruction → structured intent (e.g., `{"action": "split_column", "source": "Full Name", "targets": ["First Name", "Last Name"], "delimiter": " "}`) via the LLM
- Always return a **preview** of the resulting change (sample of affected rows, before/after) before actually applying — never execute a NL command blind
- If the LLM's parsed intent is ambiguous or low-confidence, return a clarifying question instead of guessing

## 6. Frontend UX Requirements

- **Upload → Structure Preview → Review Queue → Export** as the primary flow, completable in under a minute for a simple file
- `DataGrid.tsx`: virtualized rendering (only visible rows in DOM), frozen header row, column resize, sort/filter, before/after toggle

- `IssuesReport.tsx`: read-only exportable summary of every flagged issue and its resolution
- `RecipePanel.tsx`: shows the ordered recipe steps taken so far, with an "apply this recipe to a new file" upload target
- `NLCommandBar.tsx`: free-text input, shows a preview panel before confirming
- Loading states must describe the actual operation ("Scanning column types…", "Checking 4,213 rows against validation rules…"), never a generic spinner
- Non-destructive: all edits are undoable; keep an in-memory undo/redo stack of recipe steps
